服务平台之技术栈选择

2019-12-19
JAVA 思路

文章目录

想清楚大概结构之后，就可以选取合适的技术栈了。因为合适的技术栈可以解决很多已知问题，减少代码的开发量。如果自己开发，没有经历过大量的使用场景，肯定会有很多隐含的问题。

Redis

Redis 速度很快，因为是内存操作且使用了多路复用 I/O，所以我们可以使用 Redis 做分布式锁，控制大部分的并发情况，减少数据库的压力，并发的强控制最终是依赖数据库乐观锁，在表里加 version 字段。此处说 Redis 控制大部分并发而不是全部，是因为 Redis 分布式锁的不可靠性。

  • Redis 分布式锁的 Key 不设置有效期：有可能导致锁无法被释放。比如获取到锁的服务宕机。

  • Redis 分布式锁的 Key 设置有效期：获取到锁的服务的执行时间无法确定，所以该有效期只是一个估值。如果获取到锁的服务执行时间过长，超过了估值，会导致锁过期自动释放，此时其他服务也能获取到锁。这个问题有一种解决方案是另起线程续锁，当时间到某个阀值时自动向 Redis 续锁。但是这种解决方案也不完美，因为可能因为各种原因导致没续上，锁就已经过期。比如 GC、时钟同步（Redis 的有效期强依赖系统时钟，如果系统时钟同步，可能会导致 Key 更快或更慢失效）、没有分到 CPU 时间片等等。

MySQL

MySQL，开源，免费，支持事务。

MySQL 数据库的隔离级别默认是 RR （Repeatable Read） ，因为 RR 隔离级别是可重复读，如果在数据库中使用版本号做乐观锁强控制并发，在事务 A 中读取记录，得到一个版本号，但在正式修改之前，这个记录被事务 B 修改并提交了，导致记录的版本号增一，事务 A 再修改记录时，由于版本号不匹配，修改失败，如果事务 A 不提交或回滚，将无法读取到记录的新版本号。解决方案是使用当前读的方式查询数据（for update 或 lock in share mode），或者隔离级别修改为 RC （Read Committed）。

RocketMQ

用于解耦服务、异步执行，还有个作用是削峰填谷，不过这个需要达到一定的量级。还支持事务消息。

场景之一：支付成功之后，交易系统发消息通知对支付成功感兴趣的服务，包括但不限于：订单系统、业务系统。如果没有消息中间件，就得自己保证异步通知不丢。

RocketMQ 是阿里开源的。阿里出品，必属精品。

从生产者的角度而言，消息还有丢失的可能性，比如消息发送要与事务的操作同时成功，如果不作特殊处理，那么消息发送要么是在事务提交前，要么是在事务提交后，前者有可能消息发送成功，服务宕机，最终事务回滚，但是消息已经被发出；后者可能事务提交后，服务宕机，消息发送失败。不过这个可以通过 RocketMQ 的事务型消息解决：在事务提交前，生产者向 Broker 发送事务型消息，此时消费者拉取不到这条消息；在事务提交后，生产者向 Broker 发送确认消息，此时消费者才能拉到该消息；由于确认消息可能发送失败，所以生产者需要提供事务回查的实现， Broker 会定时发送消息回查事务是否成功。其原理类似 2PC，生产者发送事务型消息相当于 prepare，生产者发送确认消息相当于所有的 prepare 成功之后的 commit，消息回查可以根据事务的成功与否进行 commit 或者 rollback。

从 Broker 角度而言，可以通过配置双同步（主到磁盘是同步刷盘，主到从是同步复制）的方式保证 Broker 的消息不丢失。

从消费者角度而言，先消费后提交，可以保证消息至少被消费一次。但是有可能重复消费，所以需要消费者在消费时提供幂等。比如使用防重表或者业务上规避。

RPC

可以选择阿里开源的 Dubbo 框架。阿里出品，必属精品。我们公司使用的是自定义的 RPC 框架（可以支持 C/C++、iOS、Android、Java 的互调）。

在我的理解中， HTTP 请求其实就是一种特殊的 RPC ，流程都类似，客户端访问 URL （RPC 里是服务名称+接口名称+方法名称），根据域名（RPC 里是服务名称）从 DNS（RPC 里是注册中心）获取到服务器 IP 地址，将请求报文（RPC 里也是请求报文）发送到目标机器，目标机器的服务收到请求，解析请求里的路径（RPC 里是接口名称+方法名称），然后解析请求报文调用到对应的方法上，方法执行完之后，将响应报文（RPC 里也是响应报文）发给客户端，客户端解析响应报文进行渲染或其他处理。

为什么不使用 HTTP？

  • HTTP 是文本协议，报文偏长，RPC 可以自定义传输协议，使得报文更加紧凑，当然这样也会导致可读性不好，但是为了性能，这个可以接受。

  • HTTP 的服务发现、服务熔断是依赖 DNS，如果要做到类似 RPC 的服务发现、服务熔断，需要自己造轮子，实现 DNS 服务器，现有的成熟解决方案较少。

其他的 HTTP 的连接、双工问题，HTTP/2.0 已经都解决了，此处就不细说了。
